• Отличный подход. Делаем поэтапно, с узкими задачами и отдельными промптами.

  1) Frontend: шаблон или с нуля
  Рекомендую для MVP: взять минимальный шаблон и адаптировать, не с нуля.
  Причина: быстрее получить рабочий flow и меньше инфраструктурных ошибок.

  Оптимально:

  1. Vite + React + TypeScript
  2. UI без тяжелых библиотек на старте (чистый CSS или легкий набор компонентов)
  3. Потом точечно улучшать UX

  2) Архитектура проекта (MVP)
  Рекомендую структуру:

  resume-adapter/
    apps/
      web/                  # фронт (форма, превью, результат, download)
      api/                  # backend API
    packages/
      shared/               # общие типы, схемы, утилиты
    docs/
      prompts/              # промпты по задачам
      architecture.md

  Backend модули (apps/api/src):

  1. routes/ — /adapt, /health
  2. parsers/ — pdf.ts, docx.ts, txt.ts
  3. llm/ — провайдер(ы), retry, таймауты
  4. prompting/ — сборка промпта + шаблоны
  5. transform/ — нормализация и проверка ответа
  6. pdf/ — генерация ATS-friendly PDF
  7. security/ — временные файлы, автоудаление, policy

  3) Отдельные промпты под каждую задачу
  Ниже готовые промпты для агента (по одному за раз).

  1. Инициализация проекта

  Создай каркас monorepo: apps/web (Vite React TS), apps/api (Node TS + Fastify/Express), packages/shared.
  Добавь базовые npm scripts, .env.example, README.
  Ничего лишнего: без авторизации, без БД, без деплоя.

  2. Frontend: форма загрузки

  Сделай в apps/web одну страницу MVP:
  - Upload PDF/DOCX/TXT
  - Текстовое поле вакансии
  - Кнопка "Адаптировать резюме"
  - Валидация обязательных полей
  Без результата и без сложного дизайна.

  3. Backend: прием запроса и парсинг файла

  Сделай endpoint POST /adapt:
  - multipart: resumeFile + jobDescription
  - парсинг PDF/DOCX/TXT в plain text
  - унифицированный объект input
  Верни пока mock-ответ, без вызова LLM.

  4. Prompt builder

  Реализуй модуль prompt builder:
  - вход: resumeText, jobDescription
  - выход: строгий промпт для ATS-адаптации (без выдумывания фактов)
  Добавь unit-тесты на формирование промпта.

  5. LLM integration

  Подключи провайдер LLM (через абстракцию):
  - timeout, retry, обработка ошибок
  - возврат адаптированного текста
  Не делай пока fallback между несколькими провайдерами.

  6. Проверка “не выдумывать факты”

  Добавь пост-обработку ответа LLM:
  - базовая валидация дат, компаний, должностей против исходного резюме
  - если подозрение на галлюцинацию: вернуть предупреждение + сырой результат

  7. Генерация PDF

  Сделай генерацию ATS-friendly PDF из адаптированного текста:
  - 1 колонка
  - стандартные заголовки
  - системный шрифт
  Верни файл на скачивание.

  8. Страница результата

  Сделай экран результата:
  - оригинал vs адаптированный текст
  - подсветка изменений
  - кнопка Download PDF
  - блок keywordsUsed и optional matchScore

  9. Privacy/Security

  Добавь privacy-слой:
  - временные файлы с автоудалением
  - запрет долговременного хранения резюме
  - понятные ошибки без утечки чувствительных данных

  10. Минимальные тесты e2e

  Сделай минимальный e2e сценарий:
  upload -> parse -> prompt -> llm(mock) -> pdf
  Проверь успешный путь и 2 error-case (битый файл, пустая вакансия).

  4) Как работаем дальше
  Твоя мысль правильная: описание — это направление, не догма.
  На каждом шаге будем выбирать лучший вариант по критериям: скорость, риск, стоимость API, качество результата.



  • Pipeline Шага 1 (ATS-адаптация), без кода

  1. Пользователь логинится (Supabase Auth).
  2. Загружает резюме (pdf/docx/txt) и вставляет текст вакансии.
  3. Frontend отправляет POST /adapt в backend (multipart + jobDescription).
  4. Backend валидирует вход:

  - файл есть, формат разрешен, размер в лимите;
  - текст вакансии не пустой.

  5. Backend извлекает текст из файла:

  - PDF parser / DOCX parser / TXT reader;
  - если текст пустой или битый файл, возвращает понятную ошибку.

  6. Backend нормализует текст резюме и вакансии:

  - убрать мусорные символы, лишние пробелы, служебные артефакты.

  7. Backend формирует системный промпт ATS:

  - 1 колонка, стандартные заголовки;
  - ключевые слова вакансии;
  - строго не выдумывать факты, даты, компании, роли.

  8. Backend вызывает LLM API (таймаут + retry 1-2 раза).
  9. Backend валидирует ответ модели:

  - есть обязательные секции (Summary, Experience, Skills);
  - нет явных выдумок (базовая сверка дат/компаний/ролей с исходным текстом).

  10. Backend формирует ATS-friendly PDF (одноколоночный шаблон).
  11. Backend возвращает frontend:

  - adaptedText
  - pdf (или ссылка на временный файл)
  - keywordsUsed (опционально)
  - matchScore (опционально, если добавим позже).

  12. Backend пишет в Supabase только метаданные:

  - user_id, статус, время, токены/стоимость, ошибки;
  - исходное резюме и вакансию не храним постоянно (или TTL-удаление).

  SLA и ограничения MVP

  1. Цель по времени: до 30 секунд на запрос.
  2. Ограничение free: 3-5 адаптаций.
  3. Ошибки должны быть диагностируемыми (парсинг, лимит, LLM timeout, PDF fail).

  Что важно зафиксировать перед реализацией

  1. Лимит размера файла (например 5-10 MB).
  2. Храним ли PDF в Supabase Storage или сразу отдаем без хранения.
  3. Free-лимит: 3 или 5.
  4. LLM для старта: OpenAI или Anthropic.



  ПОСТЫ 

  ⏺ Тема 1: Подключение базы данных

  Сегодня подключил Supabase к своему pet-проекту — адаптатору резюме. Казалось бы, просто скопировать ключи и всё работает — но
  оказалось, что нужно разобраться в разнице между anon key и service_role key, и почему один хранится на фронте, а другой только
   на сервере. Настроил RLS-политики (Row Level Security) — это когда пользователь видит только свои данные, даже если технически
   они в одной таблице. Простая вещь, но заставила реально понять как работает безопасность на уровне БД.

  ---
  Тема 2: Security audit перед тем как двигаться дальше

  Перед тем как добавлять новые фичи, решил провести аудит безопасности. Нашёл несколько проблем: .env файл с ключами не был
  защищён в .gitignore, не было лимита на размер входящего текста, отсутствовали security headers. Добавил helmet.js
  (автоматически выставляет ~12 защитных заголовков), rate limiting (20 запросов в 15 минут с одного IP) и валидацию MIME-типов
  файлов. Главный урок: безопасность дешевле закладывать с самого начала, чем потом переделывать.

  ---
  Тема 3: Парсинг файлов — почему это нетривиально

  Пользователь загружает PDF с резюме — но сервер получает не текст, а набор байтов. Чтобы передать содержимое в GPT, нужно
  сначала извлечь текст. Подключил две библиотеки: pdf-parse для PDF и mammoth для DOCX, TXT читается нативно. Первая попытка
  провалилась — поставил pdf-parse v2 с полностью переработанным API, пришлось откатиться на стабильную v1. Небольшой, но
  показательный момент: всегда проверяй что ставишь, а не просто npm install.

  Тема 4: Подключение OpenAI — когда всё сломалось перед финишем

  Интеграция с GPT казалась финальным шагом — добавил SDK, написал 
  промпт, запустил. Но приложение отвечало одним и тем же: "Не
  удалось адаптировать". Проблема оказалась не в коде ИИ, а в      
  инфраструктуре: Node.js не загружал .env файл при запуске через
  tsx, и API-ключ просто не доходил до сервера. Урок: когда что-то
  не работает, сначала проверяй окружение, а не логику.

  ---
  Тема 5: Генерация Word-документа — UX важнее технологии

  Когда адаптация заработала, встал вопрос: в каком формате
  отдавать результат? Остановился на DOCX — большинство людей всё
  равно отправляют резюме в Word. Первая версия выглядела как
  технический дамп: разные размеры шрифтов, синие заголовки от
  встроенных стилей Word, блок с ключевыми словами внизу как
  артефакт отладки. Потратил время на единый стиль — Calibri,
  чёткая иерархия, правильные отступы между секциями. Вывод:
  скачиваемый файл — это тоже интерфейс.

  ---
  Тема 6: Тёмная тема и мелкие UX-детали, которые меняют ощущение
  продукта

  После того как основной флоу заработал, занялся деталями: тёмная
  тема (автоматически по системным настройкам), кнопка копирования
  текста с анимацией, счётчик символов у поля вакансии. Каждая из
  этих вещей — час работы. Но именно они отделяют "работает" от
  "приятно использовать". Небольшой продукт без инвестора — это
  повод сделать детали хорошо, потому что больше некому.
